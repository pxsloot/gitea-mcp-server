# Tool Annotations in Gitea MCP Server

## Overview

Tool annotations provide machine-readable metadata about MCP tools, enabling better discovery, safety, and usage guidance. The Gitea MCP Server automatically generates these annotations for all tools derived from the Gitea OpenAPI specification.

## Annotation Fields

### Title
- **Type**: `string`
- **Description**: Human-readable name for the tool
- **Source**: Generated from the OpenAPI `operationId` (camelCase â†’ snake_case by our converter), then converted to Title Case with domain prefix stripping. The `summary` field is **not** used â€” it serves as the tool's MCP `description` instead.
- **Examples**: `"Create Issue"`, `"List Pull Requests"`, `"Delete Repository"`, `"Get Organization"`

### Tags (Categories)
- **Type**: `set[string]`
- **Description**: Categorization tags for grouping and filtering tools
- **Values**: One of:
  - `repository` - Repository operations (create, delete, update, branches, contents, etc.)
  - `issue` - Issue operations (create, list, comment, label, etc.)
  - `pull_request` - Pull request operations
  - `user` - User-specific operations (keys, profile, etc.)
  - `organization` - Organization management
  - `admin` - Administrative operations
  - `misc` - Miscellaneous operations that don't fit elsewhere
- **Derivation**: Based on the OpenAPI path pattern

### readOnlyHint
- **Type**: `boolean`
- **Description**: Indicates the tool only reads data without modifying server state
- **Values**:
  - `true` - Tool does not modify state (GET, HEAD, OPTIONS methods)
  - `false` - Tool may modify state (POST, PUT, PATCH, DELETE)
- **Use case**: MCP clients can filter or warn before executing non-read-only tools

### destructiveHint
- **Type**: `boolean`
- **Description**: Indicates the tool can destroy or delete data
- **Values**:
  - `true` - Tool can delete data (currently only DELETE methods)
  - `false` - Tool does not destroy data
- **Use case**: Highlights potentially dangerous operations requiring confirmation

### idempotentHint
- **Type**: `boolean`
- **Description**: Indicates whether calling the tool multiple times with the same parameters has the same effect as calling it once
- **Values**:
  - `true` - Idempotent operations (GET, PUT, DELETE, HEAD, OPTIONS)
  - `false` - Non-idempotent operations (POST, PATCH)
- **Use case**: Agents can safely retry idempotent tools on failure; clients can optimize caching

### openWorldHint
- **Type**: `boolean`
- **Description**: Indicates whether the tool interacts with the external world (outside the MCP server)
- **Values**:
  - `true` - API tools and the `read_resource`/`call_tool` synthetic tools (they call the Gitea API)
  - `false` - Local synthetic tools (`search`, `search_tools`, `tool_info`, `list_resources`, `search_resources`, `search_docs`, `read_doc`) that operate entirely on in-memory data
- **Use case**: Helps clients understand which tools have side effects beyond the conversation

## How Annotations Are Set

Annotations are automatically generated during server initialization by the `_customize_metadata()` function in `server_setup/mcp_builder.py`:

1. **Title** is generated by `generate_tool_title()` using the OpenAPI `operationId` (converted to snake_case). The helper `_snake_to_title()` strips redundant domain prefixes and formats the remaining words. The OpenAPI `summary` field is **not** used for the title â€” it remains available as the tool's MCP `description`.
2. **Tags** are determined by `categorize_tool()` analyzing the path
3. **Hints** (`readOnlyHint`, `destructiveHint`, `idempotentHint`, `openWorldHint`) are inferred by `add_inferred_hints()` based on HTTP method semantics.
   Synthetic tools set `openWorldHint` explicitly at registration time:
   - **SAFE_METHODS** = `{GET, HEAD, OPTIONS}`
   - **DESTRUCTIVE_METHODS** = `{DELETE}`
   - **IDEMPOTENT_METHODS** = `{GET, PUT, DELETE, HEAD, OPTIONS}`

**Important**: Existing annotation values are preserved. The inference only sets a hint if its current value is `None`, allowing manual customization if needed.

## Using Annotations

### For MCP Clients

Clients can use annotations to enhance UX:

```python
# List all read-only tools
read_only_tools = [t for t in tools if t.annotations.readOnlyHint]

# Warn before destructive operations
if tool.annotations.destructiveHint:
    confirm("This operation can delete data. Continue?")

# Show tool badges/tags
if tool.annotations:
    print(f"[{tool.annotations.title}]")
    for tag in tool.tags:
        print(f"  #{tag}")
```

### For Agents

Agents can use annotations for better tool selection and error handling:

```python
# Find tools that create issues (non-read-only, non-destructive)
create_tools = [t for t in tools 
               if not t.annotations.readOnlyHint 
               and not t.annotations.destructiveHint
               and "issue" in t.tags]

# Retry logic for idempotent tools
if tool.annotations.idempotentHint:
    # Safe to retry on failure
    retry_on_failure(tool)
```

### For Developers

When extending or customizing tools, you can:

- **Set custom hints**: Provide a `ToolAnnotations` instance with predefined values. The inference will preserve them.
- **Add custom tags**: Tags added before customization will be preserved alongside the auto-generated category.
- **Inspect**: Use `tool_info(name)` to explore tool metadata.

## Technical Details

### Source Code

The annotation helpers live in `gitea_mcp_server/tools/customize.py`, orchestrated
from `_customize_metadata()` in `server_setup/mcp_builder.py`:

- `_customize_metadata(route, component, openapi_spec=...)` - Main customization entry point (in `server_setup/mcp_builder.py`)
- `add_inferred_hints(route, annotations)` - Infers hint values from HTTP method
- `generate_tool_title(route)` - Creates human-readable titles
- `categorize_tool(path)` - Determines resource category

### Testing

Unit tests in `tests/unit/test_tool_customize.py` (covering the functions in `tools/customize.py`):

- `TestInferredHints` - Verifies hint inference for all HTTP methods
- `TestCategorizeTool` - Path-to-category mapping
- `TestGenerateToolTitle` - Title generation edge cases
- `TestCustomizeMetadata` (in `tests/unit/test_mcp_builder.py`) - `_customize_metadata` tests
- Helper tests remain in `tests/unit/test_tool_customize.py` (categorize, title, hints, etc.)

**Integration tests** in `tests/integration/test_tool_annotations.py` verify the full transform chain:
annotation inference â†’ namespace â†’ extension metadata â†’ exclusion â†’ final tool metadata.
34 tests cover all hints (`readOnlyHint`, `destructiveHint`, `idempotentHint`, `openWorldHint`)
and all 7 category tags end-to-end through a real server instance.

Run tests:


## Future Enhancements

Potential improvements:

1. **Enhanced destructiveHint**: Detect destructive semantics in certain POST/PATCH operations by analyzing `operationId` keywords (e.g., "delete", "remove", "purge")
2. **Parameter-level annotations**: Mark specific parameters as sensitive (e.g., passwords, tokens) for better logging/security
3. **Response schema exposure**: Include richer output schema descriptions in annotations
4. **Custom overrides**: Support OpenAPI extensions (`x-mcp-*`) to manually override automatic inference
5. **Deprecation hints**: Mark deprecated tools based on OpenAPI `deprecated` flag
